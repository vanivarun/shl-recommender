# Approach Document: SHL Conversational Assessment Recommender

**Stack:** FastAPI · FAISS (`IndexFlatIP`) · TF-IDF (scikit-learn) · Groq API (Llama 3.1 8B Instant) · Render.com

## 1. Design Choices

**Stateless API.** The spec requires the full conversation history on every `POST /chat` call, so we keep no server-side session state. Each request is a self-contained replay: the agent re-derives context (role, seniority, skills, constraints, current shortlist) from the message history on every turn. This trades a small amount of recomputation for simplicity and horizontal scalability — important since we don't control how the evaluator load-balances requests on Render's free tier, and cold starts mean any given call may hit a fresh process.

**FAISS `IndexFlatIP` for retrieval.** Our scraped catalog of Individual Test Solutions has 338 assessments — small enough that exact search is cheap and an approximate index would only add complexity for no real speed gain. We use `IndexFlatIP` (exact inner-product search) over L2-normalized vectors, which is mathematically equivalent to cosine similarity ranking. The index is built once at process startup from the scraped catalog JSON and held in memory for the life of the process.

**TF-IDF (scikit-learn) for vectorization.** We chose classical TF-IDF over neural embeddings given the catalog size and time budget. `TfidfVectorizer` is fit once on the 338 catalog records at startup, producing sparse vectors that we densify and L2-normalize before loading into FAISS. This has near-zero inference cost, no model download, and no GPU/CPU tradeoff to manage — a meaningful advantage on Render's free tier where cold-start time and memory are both constrained. The tradeoff is that TF-IDF is a lexical/keyword-overlap method, not a semantic one (see Section 5).

**Groq API with Llama 3.1 8B Instant as the LLM.** Used for conversation orchestration — intent parsing, clarification questions, refinement logic, comparison synthesis, and refusal handling. Groq's inference speed matters given the 30-second per-call timeout: 8B Instant returns generation in well under a second, leaving margin for retrieval and validation in the same request. The LLM never invents catalog data; it only reasons over retrieved candidates and structures output into the required JSON schema.

**Separation of retrieval and generation.** FAISS/TF-IDF decides *which* catalog items are candidates; Llama decides *whether* to ask, recommend, refine, or refuse, and how to phrase the reply. The model is never asked to recall a URL from memory, only to select from a passed-in candidate list — our primary defense against hallucination.

## 2. Retrieval Setup

**Scraping.** We crawled the SHL product catalog page, filtered to Individual Test Solutions (excluding pre-packaged Job Solutions), and normalized each entry into a record with: name, catalog URL, test type code (e.g., K = Knowledge & Skills, P = Personality), and description. This scraped JSON is the *only* source recommendation URLs can come from.

**Vectorization.** We fit a single `TfidfVectorizer` on the corpus of 338 catalog records, weighting each assessment's name 3x in the indexed text (to boost exact/partial name matches), with `ngram_range=(1,2)` for phrase-level matching and English stop-word removal. The fitted vectorizer transforms both the catalog (once, at startup) and each incoming query (per turn) into the same TF-IDF space, L2-normalized before insertion into `IndexFlatIP`.

**Query construction.** On each turn, we build a rolling "requirement summary" from the whole conversation (role, seniority, skills mentioned, test-type preferences volunteered or refined) rather than vectorizing the last message in isolation. This is what makes "Actually, add personality tests" work as a refinement rather than a reset — the new constraint is merged into the summary before the next query, preserving prior context.

**Search + filtering.** We retrieve the top nearest neighbors from the 338-item index, then apply any hard constraints the user has stated explicitly as a post-filter before handing the candidate list to Llama for final selection and phrasing into 1–10 recommendations.

## 3. Prompt Design

A single system prompt with explicit branching instructions for the four required behaviors, plus retrieved candidates injected per turn:

- **Clarify:** If the conversation lacks a role/skill area and at least one of seniority or test-type preference, the agent asks exactly one targeted follow-up question and returns `recommendations: []`. Capped at one clarifying question total, and forced to recommend by the second user turn, to respect the 8-turn conversation cap.
- **Recommend:** Once sufficient context exists, the agent selects 1–10 items *only* from the retrieved candidate list, with name/URL/test_type taken verbatim from scraped JSON — never generated by the LLM.
- **Refine:** New constraints amend the standing requirement summary rather than starting a new conversation; the index is re-queried with the updated summary.
- **Compare:** For "what's the difference between X and Y" queries, we retrieve the specific catalog records by name lookup and instruct the model to answer strictly from their scraped descriptions/test types.
- **Scope guard:** A standing instruction refuses general hiring/legal advice and prompt-injection attempts, returning a short redirect and `recommendations: []`.

Output is constrained to a JSON schema, and we validate/repair the response server-side (stripping any URL not present in our catalog JSON, capping at 10 items) as a second line of defense against hallucination.

## 4. Evaluation Approach

We built a local harness (`eval_traces.py`) that replays each of the 10 provided conversation traces turn-by-turn against our live deployed `/chat` endpoint, then computes Recall@10 against each trace's labeled expected shortlist. This tests the deployed system end-to-end — network, cold starts, and all — not just the retrieval logic in isolation, which surfaced real production issues a pure unit test would have missed (see Section 5).

## 5. What Didn't Work / How We Measured Improvement

**Baseline run:** Mean Recall@10 = 0.254, with 2 of 10 traces failing outright on 30-second timeouts and 3 more timing out mid-conversation. Since the assignment's own evaluator enforces the same 30s-per-call limit, this was a genuine reliability risk, not just a test artifact.

**Diagnosis 1 — retrieval blind spot.** Our initial TF-IDF retriever matched on raw lexical overlap only. Queries like "senior Rust engineer" or "AWS/Docker backend developer" never surfaced catalog items whose names didn't share literal query terms — most notably OPQ32r (personality), which appeared in the golden shortlist of 6 of 9 successfully-answered baseline traces but was never retrieved as a candidate.

**Diagnosis 2 — latency.** Timeouts occurred mid-conversation, not only at cold start, pointing to prompt/context size rather than cold start alone.

**Fixes applied:** Reweighted the TF-IDF index (3x name weighting, bigrams, stop-word filtering) to improve match quality for paraphrased or partially-overlapping queries, and trimmed candidate count and prompt size to reduce per-call latency.

**Result:** Mean Recall@10 improved to 0.466 (from 0.254), an ~83% relative gain, with zero timeouts or errors on the final evaluation run across all 10 traces.

**Remaining weakness:** TF-IDF still misses conceptually-related but lexically-distinct items — the hardest remaining case was a multi-skill technical role (Docker, Spring, SQL, AWS all requested in one trace), where several similarly-scored skill assessments compete for a limited candidate slate and only the most lexically obvious ones surface (Recall@10 = 0.14 on that trace). A semantic embedding retriever (e.g. sentence-transformers) would likely close this gap further — the first thing we'd try with more time. We also did not have time to tune the clarify-vs-recommend threshold against the unseen holdout traces, so some persona conversations may still recommend one turn earlier or later than ideal.
